## 1. 設計強度 (Design Robustness)

アーキテクチャおよび設計の堅牢性を評価する。

| 評価 | 観点 | 所見 |
|------|------|------|
| B | 障害分離 | モノリスアーキテクチャだがモジュール境界は明確。グローバルエラーハンドラで未処理例外を捕捉。ただしモジュール障害が他モジュールに波及する構造 |
| B | 冪等性 | DB の UNIQUE 制約で重複作成を防止 (groupMembers, votes 等)。リフレッシュトークンローテーションも冪等。ただし POST エンドポイントに冪等キーの仕組みなし |
| B | 入力バリデーション | 必須フィールドチェック・型チェックは各ルートで実施。テーブル名の正規表現バリデーションあり。ただしスキーマバリデーションライブラリ (Zod 等) 未使用で、文字列長・形式の統一的検証なし |
| A | エラーハンドリング | 全エンドポイントに try-catch。適切な HTTP ステータスコード使用 (400/401/403/404/409/500)。空の catch ブロックなし。グローバルエラーハンドラ設置済み |
| B | リトライ・タイムアウト設計 | Webhook 配信: 5段階リトライ [10s, 30s, 2m, 10m, 1h]。Redis: 最大3回リトライ、指数バックオフ。PostgreSQL: 最大10回リトライ。ただし Notion/GitHub API 呼び出しにタイムアウト未設定 |
| B | 状態管理の明確性 | DB を単一の真実の源泉として使用。Redis はセッションキャッシュ。状態遷移 (予約: pending→confirmed、タスク: open→closed) は明確。ただし分散ロックなし |

### チェック項目

- [ ] 単一障害点 (SPOF) が存在しないか: **DB がSPOF。Redis 障害時は DB フォールバックあり (セッション)。ただし DB 障害時の縮退動作なし**
- [x] 外部サービス障害時の縮退動作が定義されているか: **Google Calendar 同期失敗時はローカルデータで動作可能。Webhook 配信失敗時は5回リトライ後にエンドポイント自動無効化。適切**
- [ ] 入力値の境界値・異常値に対する防御が十分か: **一部のみ (limit パラメータに上限設定 `Math.min(200, ...)`)。文字列長制限は未統一**
- [x] エラー発生時にシステムが安全な状態に遷移するか: **try-catch で適切にハンドリング。DB トランザクションは ORM レイヤーで管理。適切**
- [ ] 非同期処理のタイムアウトとキャンセル機構があるか: **Webhook 配信に 30s タイムアウト (AbortSignal)。ただし外部 API (Notion, GitHub) にタイムアウトなし**
- [ ] 競合状態 (race condition) のリスクが排除されているか: **施設予約の同一スロット二重予約を UNIQUE 制約で防止。ただしアプリ層での楽観的ロック・分散ロックなし。同時予約リクエストでの競合リスクあり**

## 2. 設計思想の一貫性 (Design Philosophy Compliance)

プロジェクトの設計思想・規約から逸脱しているコードを検出する。

| 該当箇所 | 逸脱内容 | 本来の設計思想 | 推奨修正 |
|----------|---------|--------------|---------|
| `modules/school/facility-booking/routes.ts:22` | notification モジュールの内部関数 `emitEvent` を直接インポート | モジュール間は疎結合であるべき | イベントバスのインターフェースを `src/shared/` に切り出し、モジュール間の直接依存を排除 |
| `modules/machina/webhook-handler.ts:18` | 同上: notification モジュールの `emitEvent` を直接インポート | 同上 | 同上 |
| `src/db/repository.ts` (2257行) | 全テーブルのリポジトリを単一ファイルに集約 | CLAUDE.md でリポジトリパターンを規定しており、責務ごとの分離が設計意図 | ドメインごとにファイル分割: `user-repo.ts`, `curriculum-repo.ts`, `pm-repo.ts` 等 |
| `modules/schedule/routes.ts` (1483行) | 7つの異なるエンティティ (学科/講師/カリキュラム/配置/スロット/スケジュール/教室) のルートを単一ファイルに集約 | 各モジュールは明確な責務を持つ | エンティティごとにルートファイルを分割 |
| `frontend/src/lib/api.ts` (1682行) | 17個の API クライアントオブジェクトを単一ファイルに集約 | フロントエンドもモジュール対応表に従い分離すべき | `api/calendar.ts`, `api/group.ts` 等に分割 |

### チェック項目

- [x] レイヤー間の依存方向が規約通りか: **routes → repository → DB の依存方向は一貫。ルートハンドラから直接 `db` を操作していない (CLAUDE.md 規約遵守)。適切**
- [x] 命名規則がプロジェクト全体で統一されているか: **API エンドポイント: RESTful パターン統一。変数: camelCase。DB カラム: snake_case。一貫している**
- [x] 共通パターン (リポジトリパターン, サービス層等) が一貫して適用されているか: **全テーブルにリポジトリ関数あり。直接 DB アクセスなし (admin db-viewer 除く)。適切**
- [ ] 既存のユーティリティ・ヘルパーを無視した再実装がないか: **`getUserId()` ユーティリティは全モジュールで統一使用。ただし認証チェックが `requireRole()` と手動 JWT 検証の2パターン混在 (id-service routes.ts 内)**
- [ ] 責務の配置がアーキテクチャの意図と合致しているか: **概ね適切だが上記の巨大ファイル (repository.ts, schedule/routes.ts, api.ts) が SRP 違反**
- [x] 設定値のハードコーディングがないか: **主要定数は `src/shared/constants.ts` に集約。期間時間・曜日ラベル・イベント名等。ただし OAuth URL 等一部ハードコード残存**

## 3. モジュール分割度 / 機能的凝集度 (Cohesion & Modularity)

各モジュールが単一の責務に集中しているかを評価する。

| モジュール / クラス | 凝集度評価 | 所見 |
|-------------------|-----------|------|
| `src/db/repository.ts` | 論理的凝集 | 50以上のリポジトリオブジェクトが単一ファイルに集約。DB 操作という共通の「種類」でまとめられているが、ドメインは無関係 (認証/カリキュラム/通知/PM 等)。テストや変更影響範囲の観点で問題 |
| `modules/schedule/routes.ts` | 論理的凝集 | 7つの異なるエンティティ (departments, instructors, curricula, placements, slots, schedules, rooms) を「M1 モジュール」として1ファイルに集約。各エンティティは独立した CRUD 操作を持つ |
| `frontend/src/lib/api.ts` | 論理的凝集 | 17個の API クライアントが単一ファイル。「API 呼び出し」という種類でまとめられているが各ドメインは独立 |
| `frontend/src/pages/DataManagementPage.tsx` | 手続き的凝集 | 1529行。複数のデータ管理画面を1コンポーネントに集約。状態管理が複雑化 |
| `modules/calendar/` | 機能的凝集 | Google Calendar 同期 + 手動予定管理に集中。適切な凝集度 |
| `modules/holiday/` | 機能的凝集 | 休日管理・祝日計算に特化。ユーティリティ関数を他モジュールに公開。適切 |
| `modules/voting/` | 機能的凝集 | 投票イベントのライフサイクル管理に集中。適切 |
| `modules/reminder/` | 機能的凝集 | リマインダー CRUD + テキスト解析。Alexa 拡張はサブモジュールとして分離。適切 |
| `modules/pm/` | 機能的凝集 | PM タスク同期・分析・コンフリクト解決に集中。適切 |
| `modules/notification/` | 通信的凝集 | 通知設定・テンプレート・配信・イベントバスを統括。サブモジュール (webhook, Discord, Slack, LINE) で分離されているが、orchestration 層がやや肥大 |

### チェック項目

- [ ] 1つのクラス・モジュールが複数の無関係な責務を持っていないか (SRP違反): **3件の SRP 違反: `repository.ts`, `schedule/routes.ts`, `api.ts`**
- [ ] God Object / God Class が存在しないか: **`repository.ts` (2257行, 50+オブジェクト) が God Object。`DataManagementPage.tsx` (1529行) が God Component**
- [x] 結合度が不必要に高くないか: **モジュール間の直接インポートは2箇所のみ (facility-booking→notification, machina→notification)。概ね低結合**
- [x] 循環依存が発生していないか: **循環依存なし。依存は一方向**
- [x] インターフェースが適切に分離されているか (ISP): **`ReservationPlugin` インターフェースでプラグイン API を統一。`SchulaModule` でモジュール登録を統一。適切**
- [x] パッケージ・ディレクトリ構成がドメインの構造を反映しているか: **`modules/` 配下でドメインごとにディレクトリ分離。`src/` にコア機能。適切な構造**
