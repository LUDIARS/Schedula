## 1. 機能の改善提案 (Feature Improvement)

既存機能の改善案を提案する。

| 対象機能 | 改善提案 | 期待効果 | 優先度 |
|---------|---------|---------|--------|
| 認証 (OAuth コールバック) | アクセストークン・リフレッシュトークンを URL パラメータではなく、一時的な認可コード + サーバサイド交換パターンに変更 | トークン漏洩リスクの排除 (ブラウザ履歴・Referer ヘッダ・ログからの流出防止) | High |
| トークン保存 (フロントエンド) | `localStorage` から `HttpOnly Cookie` + CSRF トークン方式に移行 | XSS 攻撃によるトークン窃取を防止 | High |
| ユーザ一覧取得 (`/auth/users/list`) | `attachGroups()` 内でユーザごとにグループを個別クエリ取得している (潜在的 N+1)。グループメンバーシップを一括取得に変更 | ユーザ数増加時のレスポンス速度改善 | Medium |
| 活動ログ (`activity-logger.ts`) | ファイルベース (`appendFileSync`) から DB テーブルまたは Redis Streams に移行 | 分散環境対応、検索・フィルタリング機能の追加、メモリ安全性の向上 | Medium |
| マイグレーション (`migrate.ts`) | try-catch による ALTER TABLE から Drizzle 標準マイグレーション (`drizzle-kit`) に移行 | マイグレーション履歴の追跡、ロールバック対応、チーム間の一貫性確保 | Medium |
| 入力バリデーション | 手動バリデーションから Zod スキーマバリデーションに統一 | バリデーションの一貫性向上、エラーメッセージの標準化、型安全性の強化 | Medium |
| `extractRows()` 重複 | `db-viewer.ts` と `settings/routes.ts` の同一実装を共通ユーティリティに抽出 | コード重複の排除、保守性向上 | Low |
| フロントエンド API クライアント | `api.ts` (1682行) をドメインごとのファイルに分割 | コードの可読性・ナビゲーション性向上 | Low |

### 観点

- **パフォーマンス最適化**: ユーザ一覧の N+1 クエリパターン改善。`getRecentLogs()` のファイル全体読み込みをストリーム化
- **ユーザ体験の向上**: 入力バリデーションエラーのフィールド単位表示。OAuth エラー時の具体的ガイダンス
- **テスタビリティの向上**: リポジトリ層の依存注入対応 (テスト用モック容易化)。巨大ファイルの分割によるユニットテスト容易化
- **運用負荷の軽減**: Drizzle マイグレーション導入による手動管理の排除。構造化ログによるログ集約の自動化

## 2. 不足機能の提案 (Missing Feature Proposal)

現在の実装に対して追加すべき機能を提案する。

| 提案機能 | 必要性の根拠 | 実装優先度 | 想定影響範囲 |
|---------|------------|-----------|------------|
| レート制限 | 認証エンドポイント (`/login`, `/register`) にレート制限がなく、ブルートフォース攻撃に対して無防備。OWASP A07 該当 | High | `src/app.ts` + ミドルウェア追加。Redis ベースのカウンタ |
| CSP (Content-Security-Policy) ヘッダ | XSS 攻撃の多層防御。現在 `X-XSS-Protection` のみで CSP 未設定 | High | `src/app.ts` のセキュリティヘッダミドルウェア |
| 構造化ログ + リクエスト ID | 現在 `console.*` ベースで分散トレーシング不可。障害調査時にリクエストの追跡ができない | High | 全モジュールのログ出力箇所。Pino/Winston 導入 |
| CI 脆弱性スキャン | npm audit で5件の脆弱性が検出されているが、CI で自動検知されない | Medium | `.github/workflows/test.yml` に `npm audit` ステップ追加 |
| API ドキュメント (OpenAPI/Swagger) | 20以上のモジュールに100以上のエンドポイントがあるが、API 仕様書なし。外部 API (`/api/external`) にのみ `docs.ts` が存在 | Medium | 全モジュールのルート定義。Hono の OpenAPI ミドルウェア (`@hono/zod-openapi`) |
| メトリクス収集 | レイテンシ・エラー率・スループット等の運用メトリクスがなく、パフォーマンス劣化を検知できない | Medium | Prometheus エクスポーター追加。`/metrics` エンドポイント |
| Redis ヘルスチェック | `/api/health` は DB のみチェック。Redis 障害時に検知できない | Medium | `src/app.ts` のヘルスチェックエンドポイント拡張 |
| 外部 API タイムアウト | Notion API、GitHub API 呼び出しにタイムアウト未設定。外部サービス障害時にリクエストがハングする可能性 | Medium | `modules/pm/`, `modules/integrations/` の fetch 呼び出し |
| パスワードリセット機能 | パスワード変更は既存パスワード入力必須。パスワード忘れ時の回復手段がない (Google OAuth のみ) | Medium | 認証モジュール。メール送信機能の追加が必要 |
| SLI / SLO 定義 | サービスレベルの目標が未定義。運用品質の基準がない | Low | ドキュメント + モニタリング設定 |
| バックアップ自動化 | DB のバックアップ/リストア手順が未整備 | Low | 運用ドキュメント + cron ジョブ |
| セッション即時無効化 | トークン侵害時に特定ユーザの全セッションを即座に無効化する管理機能がない | Low | 認証モジュール + 管理画面 |

### 観点

- **入力バリデーションの不足**: Zod 等のスキーマバリデーション未導入。文字列長制限が未統一
- **エラー通知・アラートの欠如**: Webhook 5回失敗で自動無効化は実装済みだが、管理者への通知なし
- **監査ログの不足**: 操作ログはファイルベース。ログイン試行の失敗履歴は未記録
- **ヘルスチェック・死活監視の未実装**: DB のみ。Redis・外部サービスのチェックなし
- **レート制限の未実装**: 全エンドポイントで未設定
- **バッチ処理・リトライ機構の不足**: Webhook にはリトライあり。Notion/GitHub 同期にはリトライなし
- **ドキュメント・API仕様の不足**: OpenAPI 仕様なし。README にエンドポイント一覧なし
